# Review of Changes from `c8188b22` to `c701892c`

This note records the review findings for the code changes introduced in the
commit range `c8188b22` through `c701892c`, including both boundary commits.

The review focused on behavioral regressions, control-flow changes, and missing
recovery paths introduced by the runtime/menu refactor.

## Summary

The refactor largely moved the code toward the intended boundary described in
this task, but it also introduced two clear behavior regressions and one risky
semantic change in lower-level helpers.

## Findings

### 1. Textual mode now auto-approves `aicage` self-updates

Severity: high

File:
`src/aicage/runtime/menu/textual/interaction.py`

Relevant lines:

- `TextualInteraction.confirm_aicage_update()`
- `_confirm_update_aicage()`

Problem:

The Textual interaction now returns `True` unconditionally for self-update
confirmation.

That means when `main()` calls `maybe_prompt_update(__version__,
interaction.confirm_aicage_update)`, the Textual path will immediately approve
running `pipx upgrade aicage` whenever a newer version is reported by PyPI.

Before this refactor, the update path used the normal confirmation prompt and
required an explicit user choice.

Why this is a regression:

- it changes user-visible behavior
- it introduces an external side effect without confirmation
- it affects the startup flow before normal runtime execution begins

## 2. Textual overview lost the recovery path for missing stored extensions

Severity: medium

File:
`src/aicage/config/overview_selection.py`

Relevant lines:

- `_resolve_extended_image_ref()`
- `_OverviewSelectionInteraction.choose_missing_extensions()`

Problem:

Overview resolution now validates stored extensions through
`ensure_extensions_exist(..., _OverviewSelectionInteraction())`.

The interaction used in that path does not provide a recovery decision. Instead,
`choose_missing_extensions()` raises a `RuntimeError` with the message:
`Stored extensions are missing during overview resolution.`

Before this refactor, the user was prompted to choose whether to reset the
configuration or abort.

Why this is a regression:

- accepting the Textual overview can now fail hard for stale stored config
- the old path gave the user a guided recovery choice
- the new behavior depends on the user manually noticing and editing extensions
  before pressing OK

## 3. Lower-level image update helpers now default to silent approval

Severity: medium

Files:

- `src/aicage/registry/_pull_decision.py`
- `src/aicage/registry/_image_pull.py`
- `src/aicage/registry/agent_build/_refresh.py`
- `src/aicage/registry/ensure_image.py`

Problem:

The default behavior of the image-update decision helpers changed from
prompt-or-caller-supplied behavior to unconditional approval when no
confirmation callback is provided.

Examples:

- `decide_pull()` now defaults to a callback that returns `True`
- `pull_image()` now defaults to a callback that returns `True`
- `refresh_base_image()` now defaults to a callback that returns `True`
- `ensure_image()` normalizes `None` to a callback that returns `True`

The current CLI flow mostly compensates for this through `prepare_image()`, but
the module-level helper behavior itself still changed.

Why this is risky:

- direct callers no longer get the old prompt-driven behavior
- silent semantic changes at helper level are easy to miss
- the new default is more aggressive than the old one
